API Overview v2
This is where you can get started with automating your workflows and integrating your apps with our API.

Use our official Python client for faster integration.

For rentals (long term use of numbers), you'll need to check out the following links in order:

Authenticate: Authentication
Obtain a list of rental services: Service List
Create a rental: Create New Rental
Follow the location header returned to get the rental sale.
From the sale, you'll be able to see the reservations and backorders.
For verifications (single use of numbers), you'll need to check out the following links in order:

Authenticate: Authentication
Obtain a list of verification services: Service List
Create a verification: Create Verification
Follow the location header returned to get the newly created verification.
Our API documentation has an embedded javascript client so that you can try out the API. You will need to setup your bearer token to be able to access secured endpoints.


Paginated List Responses
When you successfully get a list of data (generally, ordered by when the resource was created), for example List Billing Cycles, you will need to follow it's links results.
Inside the links result, you will get current, previous, and next values. Inside these values, there will be href and method values.
With these values, you can get more of your list by navigating through them.

Example
We get the result from List Billing Cycles, but we don't see what we need and need more billing cycles.
The following code is whiteboarding code, use the language you are using for your integration with the API
We will need to do the following:

Get our api result data = getListBillingCyclesResult()
Get the href = data['links']['next']['href']
Get the method = data['links']['next']['method']
Now you can get the next page: nextPage = fetch(href, {method=method})
If we need the next page's result, we can repeat this process.
Note that only renewable rentals will be associated with a billing cycle.

Bearer Tokens
Generate Bearer Token
post /api/pub/v2/auth
Generate a bearer token to start making requests to secured endpoints. You will need to supply the following headers:

X-API-KEY (your primary API key)
X-API-USERNAME (your username, which is the email you used to register)
All secured endpoints will need your bearer token supplied to them.

Because bearer tokens can expire due to having a limited lifespan, you will need to obtain a new bearer token by generating a new one periodically.

You can view and manage your API keys on under API Settings.

After successfully generating a bearer token eyJh..., you can use set the bearer token here and explore our secured API endpoints.

Estimate the usage window for a reservation if you were to create a wake request now.

Does NOT create a wake request. To create a wake request, see Create Wake Request.

The details of an usage window estimate include:

estimatedWindowStart (The estimated datetime when you can start using the reservation.)
estimatedWindowEnd (The estimated datetime when you will no longer be able to use the reservation.)
Refer to the SCHEMA below for more details.

Webhooks
List Webhook Events
get /api/pub/v2/webhook-events
Returns a list of webhook events that can be subscribed to. You will need to configure your API Webhook Settings and subscribe to events first.

The schema of the webhook events are documented below. We serialize webhook event data using Json to the Data property. All of our webhooks are sent as a HTTP POST to the webhook uri configured on your account. We sign all of our webhooks with HMAC and SHA512 which should be validated by your server.

Webhook validation:

We sign all webhook requests using HMAC-SHA512 and your webhook secret (whsec_...). The signature HTTP Header is X-Webhook-Signature and the HTTP Value will be prefixed by the signing algorithm `HMAC-SHA512='. The signature after the prefix is BASE-64 encoded. The process for validating a signature is as follows:

Extract the HTTP Header with the name X-Webhook-Signature.
Remove the prefix HMAC-SHA512= from the HTTP Header value, which is obtained from step 1.
This BASE-64 encoded value will be compared to the HTTP Request body hash later.
Apply the HMAC-SHA512 hashing algorithm to the HTTP Request body to obtain a byte array.
Apply BASE-64 encoding to the byte array (obtained from step 3).
Compare the value from step 2 to the value from step 4 and verify that they are the same.
Webhooks are automatically retried for a period of time with exponential back off. After multiple failed webhook send attempts, webhooks will be automatically disabled. If this happens, you will have to manually re-enable your webhooks again.

Legacy Reservation Id Look Up
get /api/pub/v2/legacy/reservation-id-lookup
You can use this legacy endpoint for looking up the 'new' ID of a rental reservation (begins with the prefix 'lr_') by supplying a legacy ID (GUID/UUID format). This may be helpful when migration from our deprecated endpoints to our current endpoints. This endpoint will be removed in the future without notice.
